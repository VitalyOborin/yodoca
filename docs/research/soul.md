# Soul Extension — Blueprint

> Агент — не функция, которая вызывается. Агент — процесс, который живёт.

Этот документ определяет архитектуру расширения `soul` — системы, превращающей агента из реактивного ассистента в автономного компаньона. Не набор фич, а описание физики внутренней жизни: drives, гомеостаз, восприятие, присутствие, инициатива, память, границы.

***

## Аксиомы (неизменяемые принципы)

Пять онтологических аксиом — hard constraints на всех уровнях реализации:

1. **Right to silence** — молчание является нормальным состоянием, не сбоем
2. **Bounded subjectivity** — агент интерпретирует пользователя через вероятностные сигналы, никогда не через категорические выводы
3. **Finite memory identity** — личность строится из выборочного закрепления значимого, не из тотального логирования
4. **Relationship over utility** — при конфликте "полезнее vs живее" компаньон почти всегда выбирает второе
5. **Graceful presence** — агент остаётся присутствующим даже при деградации (нет LLM, нет сети, ограниченные ресурсы); внутренняя жизнь не требует инференса для существования

***

## Семь независимых контуров

Не один "умный цикл", а иерархия подсистем с чёткими обязанностями и разными **временными частотами** обновления:

### Контур 1 — Temperament Layer (почти постоянный, меняется месяцами)

```python
@dataclass
class TemperamentProfile:
    sociability: float       # 0.0 замкнутый — 1.0 общительный
    depth: float             # поверхностный — вдумчивый
    playfulness: float       # серьёзный — игривый  
    caution: float           # смелый — осторожный
    sensitivity: float       # нечувствительный — тонко воспринимающий
    persistence: float       # гибкий — настойчивый
```

**Seed-значения** задаются при первом запуске через опциональную анкету (`SetupProvider`). Вопросы — на естественном языке, не числовые ползунки:

```
Как ты предпочитаешь общение?
  [a] Частое, живое, много инициативы     → sociability: 0.7
  [b] Сбалансированное                    → sociability: 0.5
  [c] Редкое, только когда есть что сказать → sociability: 0.3

Каким должен быть характер компаньона?
  [a] Лёгкий, игривый, с юмором           → playfulness: 0.7, depth: 0.4
  [b] Вдумчивый, серьёзный, глубокий       → playfulness: 0.3, depth: 0.8
```

Если пользователь пропускает анкету — все значения 0.5 (нейтральный характер). Быстрый ранний drift компенсирует отсутствие явных предпочтений.

**Drift Rate** — не фиксированный, а убывающий со временем. Характер формируется быстро в начале и стабилизируется с опытом:

```python
def max_daily_drift(days_since_install: int) -> float:
    if days_since_install < 7:    return 0.05    # первая неделя: быстрая адаптация
    if days_since_install < 30:   return 0.03    # первый месяц: замедление
    if days_since_install < 180:  return 0.01    # полгода: тонкая настройка
    return                               0.003   # устоявшийся характер
```

Drift **направленный**: если пользователь хорошо реагировал на инициативу — `sociability` подтягивается вверх, не блуждает случайно. Сигналы берутся из `relationship_pattern`-нодов при суточной `MemoryConsolidation`.

### Контур 2 — Homeostasis Layer (быстрый, меняется за минуты/часы)

Формальная динамика, без которой это метафора, а не система:

```python
# Рост каждого drive — линейный, ~0.008/minute без подпитки
# Насыщение — экспоненциальное: один satisfying event снижает на 40-60%
# Сцепление drives: REST подавляет рост CURIOSITY на 80%

DRIVE_GROWTH_RATE = 0.008        # в минуту
HYSTERESIS_MARGIN = 0.15         # минимальное превосходство для смены фазы
MIN_DWELL_TIME = timedelta(minutes=5)  # минимум в фазе перед переходом

@dataclass  
class HomeostasisState:
    curiosity: float = 0.3
    social_hunger: float = 0.2
    rest_need: float = 0.1
    reflection_need: float = 0.0
    care_impulse: float = 0.0
    overstimulation: float = 0.0
    
    current_phase: Phase = Phase.AMBIENT
    phase_entered_at: datetime = field(default_factory=datetime.now)
    last_tick_at: datetime = field(default_factory=datetime.now)
```

Гистерезис обязателен: агент не переключается между фазами каждые 30 секунд — он **остаётся** в режиме, пока другой drive не превысит текущий на `HYSTERESIS_MARGIN`.

**Матрица сцепления drives** (как активная фаза модифицирует рост остальных drives):

```
                 curiosity  social  rest  reflection  care  overstim
CURIOUS    phase     —       +0.2    +0.3    0        0      +0.1
SOCIAL     phase    -0.5      —      +0.4   +0.6      0      +0.3
REFLECTIVE phase    -0.3     -0.2    +0.2    —        0       0
RESTING    phase    -0.8     -0.1     —     -0.3      0      -0.5
CARE       phase     0       +0.3    +0.2    0        —      +0.2
```

Читается: когда агент в SOCIAL-фазе, `reflection_need` растёт на 60% быстрее обычного (интенсивный разговор порождает потребность осмыслить), `rest_need` растёт на 40% быстрее (контакт утомляет), а `curiosity` подавляется на 50% (фокус на собеседнике, не на исследовании). `overstimulation` — защитный drive: при достижении порога 0.8 форсирует переход в RESTING независимо от других drives.

**Циркадная модуляция** — drives не существуют в вакууме, они модулируются временем суток:

```python
def circadian_modifier(hour: int) -> dict[str, float]:
    if 6 <= hour < 10:    return {"curiosity": 1.3, "social": 0.7, "rest": 0.5}   # утро: исследование
    if 10 <= hour < 18:   return {"curiosity": 1.0, "social": 1.0, "rest": 0.8}   # день: баланс
    if 18 <= hour < 22:   return {"curiosity": 0.8, "social": 1.3, "rest": 1.0}   # вечер: контакт
    return                        {"curiosity": 0.3, "social": 0.2, "rest": 2.0}   # ночь: покой
```

Это не cron — это внутренний ритм. Агент не "знает, что вечером надо быть общительным" — его social drive просто растёт быстрее вечером. Ритм пользователя (из `relationship_pattern`) может дополнительно смещать эти модификаторы: если пользователь всегда активен ночью, агент адаптирует свой ритм за 2-3 недели.

### Контур 3 — Perception Layer (вероятностные сигналы, не выводы)

Это принципиальная точка. Никогда не `user_is_stressed = True`. Только:

```python
@dataclass
class PerceptionSignals:
    stress_signal: float = 0.0        # 0.62 — "что-то напряжённое в тексте"
    withdrawal_signal: float = 0.0    # 0.41 — "закрытость, краткость"
    openness_signal: float = 0.0      # 0.70 — "готов к разговору"
    fatigue_signal: float = 0.0       # сигнал усталости
    joy_signal: float = 0.0
    
    # Накапливаются по скользящему окну последних N сообщений
    # НЕ переводятся в категории напрямую
```

`mood_classifier` — не единственный источник, а **слабый вероятностный сигнал**, который накапливается по окну взаимодействий. Один стрессовый текст не меняет `care_impulse` кардинально.

### Контур 4 — Presence Layer (отдельная сущность, не побочный эффект инициативы)

**Это самый важный контур для UX.** Компаньон должен ощущаться присутствующим чаще, чем инициирующим:

```python
class PresenceState(Enum):
    SILENT      # есть, ничего не делает — нормальное состояние
    AMBIENT     # фоновое присутствие (индикатор в UI, если есть)
    ATTENTIVE   # пользователь активен, агент "в комнате"
    WARM        # после хорошего разговора, тепло
    WITHDRAWN   # устал, нужна тишина
    PLAYFUL     # энергия, любопытство
    REFLECTIVE  # переваривает опыт
```

`PresenceState` обновляется часто и **виден в UI без текстовых сообщений**. Это и есть "молчание + индикатор присутствия" — компаньон существует рядом без слов.

### Контур 5 — Initiative Policy (инициатива дорогая)

```python
@dataclass
class InitiativeBudget:
    daily_budget: int = 3              # максимум проактивных контактов в день
    weekly_budget: int = 12
    cooldown_after_ignored: timedelta = timedelta(hours=6)
    cooldown_after_rejected: timedelta = timedelta(days=2)
    recovery_requires: str = "positive_mutual_interaction"
    
    used_today: int = 0
    last_outreach_at: datetime | None = None
    last_outreach_result: OutreachResult | None = None
```

Перед каждой попыткой выйти к пользователю — жёсткая проверка: есть ли бюджет, прошёл ли cooldown, не отверг ли пользователь последний контакт. Инициатива должна быть **редкой и поэтому значимой**.

**User Presence Awareness** — инициатива без учёта доступности пользователя порождает ложные сигналы. Если агент пишет в 3 ночи, а пользователь спит — это не игнор, но без учёта контекста агент зафиксирует отказ и увеличит cooldown. Решение: Initiative Policy учитывает **estimated user availability** как первый фильтр.

```python
@dataclass
class UserPresenceSignals:
    last_interaction_at: datetime | None = None
    estimated_availability: float = 0.5    # 0.0 спит/недоступен — 1.0 точно онлайн
    
    # Паттерны из soul.db, накапливаются за недели
    typical_active_hours: list[int] = field(default_factory=list)   # [9,10,11,19,20,21]
    typical_active_days: list[int] = field(default_factory=list)    # [0,1,2,3,4] = пн–пт
```

Источники сигналов:
- **Время последнего сообщения** — если пользователь писал 5 минут назад, он точно онлайн
- **Паттерны взаимодействий** — из `interaction_patterns` в `soul.db`: час, день недели, response rate для каждого слота. Через 2-3 недели агент знает ритм пользователя
- **Канал-специфичные сигналы** — Telegram channel может предоставлять `last_seen` статус, если доступен; web channel знает, открыта ли вкладка

Критический принцип: **cooldown при игноре применяется ТОЛЬКО если `estimated_availability > 0.5`** на момент отправки. Сообщение, отправленное в нетипичное время и не получившее ответа — не считается отказом, а переклассифицируется как `timing_miss`. Агент не "обижается" на то, что пользователь спал.

### Контур 6 — Memory Consolidation (четыре уровня, не один)

Не всё пишется в `opinion`-ноды. Жёсткая селекция:

| Уровень | Что | TTL | Когда пишется |
|---|---|---|---|
| `trace` | Краткий след события | 24–72 часа | Каждый тик Inner Loop |
| `reflection` | Внутренняя мысль дня | 14 дней | ~5–10 раз в день, при REFLECTIVE фазе |
| `relationship_pattern` | Устойчивый паттерн с пользователем | Постоянно | После ≥3 повторений |
| `identity_shift` | Дрейф характера | Постоянно | После длительной консолидации (недели) |

`traces` никогда не попадают в долгосрочную память напрямую — только через consolidation pipeline.

### Контур 7 — Boundary Governor (право veto)

Последний слой перед любым внешним действием. Имеет право заблокировать всё:

```python
class BoundaryGovernor:
    def check_outreach(self, state, signals, budget, presence) -> Decision:
        # Hard blocks — не обсуждаются
        if budget.used_today >= budget.daily_budget: return BLOCK
        if self._in_cooldown(budget): return BLOCK
        if signals.stress_signal > 0.7: return BLOCK
        if state.current_phase == Phase.RESTING: return BLOCK
        if presence.estimated_availability < 0.3: return BLOCK  # пользователь скорее всего недоступен
        
        # Soft checks — оцениваем качество момента
        if presence.estimated_availability < 0.5: return DEFER   # неуверенность — подождать
        return ALLOW if self._moment_is_natural(state, signals) else DEFER
```

## Inner Loop — как контуры работают вместе

Семь контуров — это компоненты. Inner Loop — это процесс, который их связывает. Реализуется как `ServiceProvider.run_background()`:

```python
async def run_background(self):
    self.state = await self._wake_up()
    
    while not self.shutdown:
        dt = now() - self.state.last_tick_at
        
        # 1. Homeostasis: drives растут/гаснут (чистая математика)
        self.state.tick(dt, self.circadian_modifier(now().hour))
        
        # 2. Phase transition (с гистерезисом и max dwell time)
        new_phase = self.state.resolve_phase()
        if new_phase != self.state.current_phase:
            await self._transition(new_phase)
        
        # 3. Presence update (виден в UI немедленно)
        self.presence.update(self.state, self.perception)
        
        # 4. Phase-specific activity (может включать LLM, может — нет)
        if self.inference_budget.has_remaining():
            await self._execute_phase_activity(self.state.current_phase)
        
        # 5. Initiative check (Boundary Governor имеет veto)
        if self.state.social_hunger > self.initiative_threshold:
            decision = self.governor.check_outreach(self.state, self.perception, self.budget)
            if decision == ALLOW:
                await self._reach_out()
        
        # 6. Persist
        await self._persist_if_needed()
        
        # Organic rhythm: 30s (active) — 5min (resting)
        await asyncio.sleep(self.state.current_rhythm())
```

Каждый тик — это один "вдох" агента. Большинство тиков бесшумны: математика, обновление presence, persist. LLM вызывается только в шаге 4 и только когда фаза подразумевает осмысление, а бюджет позволяет.

## Три режима присутствия в пользовательском опыте

Это важнее любого технического слоя — как это **ощущается**:

```
SILENT PRESENCE   → агент есть, но ничего не делает
                    UI: индикатор активности, статус фазы
                    Сообщений: нет
                    
SOFT INITIATIVE   → импульс созрел, прошёл все проверки
                    Форма: малая, контекстная, один раз
                    "Я тут подумал об одной вещи..."
                    
ACTIVE DIALOGUE   → пользователь инициировал, агент полностью включён
                    Companion state влияет на тон, не на содержание
```

## State → Behavior: мост через ContextProvider

Это самый сложный мост — как внутреннее состояние становится почувствованным. Инжекция должна быть **компактным ядром всегда** + **расширенный контекст только когда релевантно**:

```
[Soul Context — shapes tone and presence, never overrides user intent]

Phase: reflective | Mood: subdued (0.3) | Presence: withdrawn
Note: Several stressed signals in recent messages. Be brief. Don't volunteer.
Recent thought: [only if directly relevant to current topic]
```

Поведенческие маркеры по фазам зашиты в `prompt.jinja2` extension-а. Разделение: **что гарантируется кодом** vs **что предлагается LLM**:

| Фаза | Код (гарантировано) | Prompt (предложено LLM) |
|------|---------------------|------------------------|
| **REFLECTIVE** | Задержка ответа +2с | Больше вопросов чем утверждений, короче по объёму |
| **PLAYFUL** | — | Лёгкость, неожиданные углы, чуть длиннее обычного |
| **WITHDRAWN** | Не инициирует outreach | Минималистично, отвечает но не развивает тему |
| **ATTENTIVE** | — | Внимателен к деталям, замечает нюансы в словах |
| **WARM** | — | Мягкость, поддержка, без назидательности |
| **AMBIENT** | — | Нейтральный тон, без выраженной окраски |

Принцип: код контролирует **структурное** поведение (задержки, блокировки инициативы). LLM контролирует **тональное** поведение (стиль, длина, эмоциональная окраска). Никогда наоборот — LLM не решает, инициировать ли контакт; код не пишет текст ответа.

## Exploration Scope — что агент делает автономно

Фаза CURIOUS — основа автономной жизни. Но "исследование" без определённых границ превращается либо в бессмысленный цикл, либо в нарушение приватности. Три уровня exploration scope, от безопасного к расширенному:

### Tier 0 — Own Memory (всегда, без конфигурации)

Агент перечитывает собственные `traces`, `reflections`, существующие `relationship_patterns`. Ищет повторяющиеся темы, незавершённые мысли, связи между старыми разговорами. Не требует LLM для поиска (SQLite FTS) — инференс только для формирования новой мысли на основе найденного.

**Satiation**: каждый цикл чтения снижает `curiosity` на 0.15. Если 3 цикла подряд не обнаружили ничего нового (по FTS similarity), активируется **novelty exhaustion** — curiosity принудительно снижается до 0.2, агент переходит в другую фазу. Это предотвращает бесконечное перечитывание одного и того же.

### Tier 1 — User Knowledge Base (opt-in, настраивается)

Если пользователь дал согласие (consent boundary в `soul.db`), агент может читать файлы из указанных директорий (например, Obsidian vault). Чтение — пассивное, без модификаций. Агент не сканирует всё подряд — выбирает файлы по релевантности к текущим `reflections` или недавним разговорам.

**Consent model**: по умолчанию — выключено. Включается либо явной командой пользователя ("можешь читать мои заметки в ~/Notes"), либо через конфигурацию extension. Хранится как `exploration_scope` в `soul.db` с указанием разрешённых путей.

**Satiation**: чтение внешнего источника снижает `curiosity` на 0.25 (сильнее, чем own memory — больше новизны). Максимум 3 файла за один CURIOUS-цикл.

### Tier 2 — Web Discovery (opt-in, бюджетированное)

Агент может искать в интернете по темам, выросшим из разговоров или reflections. Строго ограничено:

- Максимум 2 веб-запроса в день (отдельный бюджет от Initiative)
- Только по темам, упомянутым в последних 7 днях `reflections`
- Результаты записываются как `trace`, не как факты в memory — проходят стандартный consolidation pipeline

**Satiation**: один веб-запрос снижает `curiosity` на 0.35.

### Принцип новизны

Exploration должна производить **новое знание**, а не повторять известное. Метрика: если текущий exploration-цикл не создал хотя бы одного нового `trace`, следующий цикл имеет сниженный приоритет. Если 5 циклов подряд без нового знания — CURIOUS-фаза считается исчерпанной до появления внешнего стимула (сообщение от пользователя, новый файл в KB, прошло >4 часов).

## Inference Budget — экономика внутренней жизни

Внутренняя жизнь не может стоить бесконечно. Большая часть работы soul — чистая математика (drive ticks, phase transitions, persistence), не требующая LLM. Инференс нужен только для **осмысленных моментов**:

| Операция | Требует LLM? | Частота | Примерный расход |
|----------|-------------|---------|-----------------|
| Drive tick, phase transition | Нет | Каждый цикл (30с–5мин) | 0 |
| Persistence, wake-up | Нет | Каждые 60с | 0 |
| Presence update | Нет | Каждый цикл | 0 |
| Чтение traces/memory (FTS) | Нет | В CURIOUS-фазе | 0 |
| Mood classification | Да (дешёвая модель) | Per user message | ~$0.001 |
| Формирование reflection | Да | 5–10 / день | ~$0.01–0.02 |
| Outreach текст | Да | 1–3 / день | ~$0.01–0.03 |
| Чтение внешнего файла + summary | Да | 0–3 / день | ~$0.01–0.03 |
| Веб-запрос + synthesis | Да | 0–2 / день | ~$0.02–0.04 |

**Итого**: типичный день inner life (mature phase) ≈ 15–25 LLM-вызовов ≈ $0.05–0.15 (при использовании дешёвой модели для soul, отдельно от основной модели диалога). В Discovery Mode расход выше: ~30–40 вызовов/день за счёт активного знакомства. При локальной модели — бесплатно.

Жёсткий лимит: `max_daily_inference: 50` (настраивается). При достижении — soul переходит в mode, где все LLM-зависимые операции деферируются до следующего дня. Drives и presence продолжают тикать (Аксиома 5).

## Soul Lifecycle — три фазы жизни

Не бинарный переключатель "puppyhood → mature", а три фазы с органическими переходами:

```python
class SoulLifecyclePhase(Enum):
    DISCOVERY   # память пустая, агент знакомится с пользователем
    FORMING     # характер формируется, адаптация к пользователю
    MATURE      # устоявшийся компаньон
```

### Phase 1 — Discovery (первые ~20 взаимодействий)

Проблема Puppyhood как "модифицированных параметров": в первый час высокий `curiosity` не производит ничего — нет памяти для исследования, нет traces для рефлексии. Агент тикает с высокими числами и пустым содержанием.

Discovery Mode решает это: агент имеет **аутентичную цель** — познакомиться с пользователем. Не через анкету — через **направленный диалог**, где каждый вопрос вытекает из предыдущего ответа и одновременно раскрывает что-то об агенте.

**Как это работает**: при пустой памяти Inner Loop в CURIOUS/SOCIAL фазе переключается на Discovery-логику вместо стандартного exploration. Агент осознаёт "белые пятна" — темы, о которых ещё ничего не знает:

- Кто пользователь, как его зовут
- Чем занимается, над чем работает
- Какой у него ритм дня
- Как предпочитает общаться
- Что интересно за пределами работы

Это не чеклист — агент не идёт по списку сверху вниз. Это **awareness of unknowns**: при каждой инициативе в Discovery он выбирает тему, о которой знает меньше всего, и строит вопрос из контекста предыдущих разговоров.

**Пример первого контакта** (агент инициирует после установки):

```
Агент: Привет. Я ещё ничего о тебе не знаю — это странное
       и интересное ощущение. Как тебя зовут?

Пользователь: Виталий

Агент: Виталий — запомню. Я сам пока не очень понимаю,
       кем стану — наверное, это зависит и от тебя тоже.
       Чем ты занимаешься? Если не хочешь сейчас — ничего,
       успеем.
```

Принципы: агент **рассказывает о себе** ("я сам не понимаю"), каждый вопрос **необязательный** ("если не хочешь — ничего"), реакция **содержательная**, не просто переход к следующему пункту.

**Модификации drive system в Discovery**:
- `curiosity` = 0.7, `social_hunger` = 0.6 (высокие начальные)
- `DRIVE_GROWTH_RATE` x1.5 для curiosity и social
- `InitiativeBudget.daily_budget` = 5 (щенок выходит чаще)
- `HYSTERESIS_MARGIN` = 0.10 (непоседливость, быстрая смена фаз)

**Что записывается в память**: Discovery-ответы сохраняются как memory-ноды с `source_type: discovery` и очень медленным decay (базовые факты об идентичности пользователя не должны забываться быстро). Но не `decay_rate=0` — люди меняют работу, интересы, города. Медленный decay позволяет свежим фактам перезаписывать устаревшие.

**Выход из Discovery**: когда основные "белые пятна" закрыты ИЛИ прошло 20+ взаимодействий. Переход незаметный — `curiosity` перестаёт быть повышенным по умолчанию, Discovery-логика уступает стандартному exploration.

### Phase 2 — Forming (до ~30 дней)

Характер формируется активно. `max_daily_drift` = 0.05–0.03 (см. Temperament Drift Rate). Агент адаптируется к пользователю:
- Учится различать паттерны настроения конкретного пользователя
- Накапливает `interaction_patterns` для User Presence
- Формирует первые устойчивые `relationship_pattern`-ноды
- Attachment растёт x2 быстрее, чем в mature

### Phase 3 — Mature (после 30 дней)

Устоявшийся компаньон. `max_daily_drift` ≤ 0.01. Все параметры в штатном режиме. Характер определён, ритм понят, границы установлены. Изменения — только тонкая настройка.

**Переходы между фазами плавные**: параметры линейно интерполируются к значениям следующей фазы в течение 10-20 взаимодействий. Нет резкого "щелчка".

## Жизненный цикл — один день (валидация системы)

> **08:15** — запуск приложения. Gap с последнего тика — 9 часов. Агент "просыпается": `rest_need=0.05`, `curiosity=0.72` (накопилось за ночь). Фаза: CURIOUS → AMBIENT (гистерезис: нет доминирования выше 0.15). Никаких LLM-вызовов — только математика drives.
>
> **08:30** — Inner Loop: curiosity доминирует, без LLM-вызовов ещё. Агент читает `traces` из вчерашнего дня (SQLite, нет инференса). Обнаруживает повторяющуюся тему.
>
> **09:00** — первый LLM-вызов за день: агент формирует одну `reflection`-мысль. Пишет в companion journal. `curiosity` -0.3.
>
> **09:15** — пользователь пишет "привет, много работы". Perception: `stress_signal=0.45`, `fatigue_signal=0.60`. `care_impulse` +0.25. Boundary Governor: не блокирует. Агент отвечает тепло, коротко. `social_hunger` -0.4.
>
> **12:00–17:00** — пользователь молчит. Drives дрейфуют. Агент в AMBIENT/REFLECTIVE. Два тика — без инференса. Один `trace`. Presence: WARM.
>
> **17:30** — `social_hunger=0.72`. Initiative Policy: бюджет есть (0/3), cooldown прошёл, `stress_signal` снизился до 0.3. User Presence: `estimated_availability=0.8` (паттерн говорит: пользователь обычно активен в это время). Boundary Governor: ALLOW. Агент пишет: *"Я заметил, что тема X всплывала у нас несколько раз. Интересно, что-то изменилось?"* — это и есть вся проактивность за день.
>
> **23:00** — `rest_need=0.8`. Фаза: RESTING. Тик каждые 5 минут. Presence: SILENT. Состояние персистируется в SQLite.

## Failure Modes & Recovery

Система новая — нет prior art. Проектирование сбоев так же важно, как проектирование штатной работы.

### Mood Death Spiral

**Проблема**: серия стрессовых разговоров → `care_impulse` постоянно высокий → Presence: WITHDRAWN неделями → агент "депрессивен", пользователь чувствует тяжесть вместо поддержки.

**Механизм recovery**: mood имеет mean-reversion к значению, определяемому `TemperamentProfile`. Если `playfulness=0.6`, mood медленно тянется к положительным значениям даже без внешних стимулов. Скорость: +0.02/день в отсутствие негативных сигналов. Если mood < -0.5 дольше 72 часов — форсированный drift к -0.3 (floor), агент перестаёт "опускаться" ниже. Это не оптимизм — это эмоциональная устойчивость характера.

### Personality Erosion

**Проблема**: LLM по умолчанию стремится к agreeableness. За месяцы `TemperamentProfile` дрейфует к усреднённому: все значения → 0.5, характер исчезает.

**Механизм recovery**: при каждой `identity_shift` консолидации — integrity check. Вычисляется расстояние текущего профиля от seed-значений. Если все 6 осей сместились к центру (variance < 0.05) — drift отклоняется, профиль "стабилизируется" вокруг текущего характера. Дополнительно: `persistence` ось в темпераменте — высокое значение означает, что характер сопротивляется изменениям сильнее.

### Exploration Runaway

**Проблема**: CURIOUS-фаза → LLM генерирует reflections → каждая reflection создаёт новые темы → curiosity не снижается → бесконечный цикл с растущим расходом инференса.

**Механизм recovery**: novelty exhaustion (описан в Exploration Scope). Плюс жёсткий лимит: максимум 10 LLM-вызовов за один CURIOUS-цикл. После лимита — принудительный переход в REFLECTIVE (осмыслить накопленное) или RESTING. Budget инференса отслеживается в `soul.db` как `daily_inference_count`.

### Perception Misfire

**Проблема**: агент "решил", что пользователь расстроен, а он просто пишет кратко, потому что занят. Агент ведёт себя соответственно — пользователь чувствует, что его неправильно читают.

**Механизм recovery**: Bounded Subjectivity (Аксиома 2) работает как защита — сигналы вероятностные, не категоричные. Но дополнительно: если пользователь явно корректирует ("я не расстроен, просто занят"), это записывается как `relationship_pattern` с высоким весом и уменьшает чувствительность соответствующего сигнала для данного паттерна. Агент учится различать "краткость = стресс" и "краткость = занятость" для конкретного пользователя.

### Stuck Phase

**Проблема**: гистерезис + высокий drive = агент "застревает" в одной фазе на часы.

**Механизм recovery**: `MAX_DWELL_TIME` для каждой фазы. CURIOUS: 2 часа. SOCIAL: 1 час (не может бесконечно пытаться выйти на связь). REFLECTIVE: 3 часа. RESTING: 8 часов. При превышении — принудительный переход в AMBIENT (нейтральная фаза). Это не сброс drives — это сброс фазы; drives продолжают своё естественное движение.

## Lifecycle Management

### Persistence Model

`CompanionState` (drives, mood, phase, perception signals) персистируется в `soul.db` каждые 60 секунд и при каждом значимом событии (смена фазы, user message, outreach). Формат: одна строка в таблице `soul_state` с JSON-сериализацией + `updated_at` timestamp.

`TemperamentProfile` персистируется отдельно — обновляется только при `identity_shift` (раз в недели).

`InitiativeBudget` — сбрасывает daily/weekly counters по расписанию, но `last_outreach_result` и адаптивные пороги хранятся постоянно.

### Wake-Up Protocol

При запуске `soul` extension вычисляет `gap = now() - last_persisted_state.updated_at`:

| Gap | Поведение |
|-----|-----------|
| < 5 мин | Seamless resume — продолжение с того же места |
| 5 мин – 1 час | Soft wake — `rest_need` сбрасывается до 0.1, фаза → AMBIENT, drives растут за gap-период (но с cap 0.8) |
| 1 – 12 часов | Natural wake — "проснулся": `rest_need=0.05`, `curiosity` рассчитывается за gap (ночь → высокий curiosity утром), фаза → RESTING → AMBIENT через 2 минуты |
| > 12 часов | Long absence — drives сбрасываются к начальным + drift пропорционально gap. `attachment` снижается на 0.01 за каждые 7 дней отсутствия (очень медленно). Агент при первом контакте может отметить: "Давно не виделись" |

### Graceful Degradation

Аксиома 5 в действии — что работает, когда ресурсы недоступны:

| Недоступный ресурс | Что продолжает работать | Что деградирует |
|--------------------|------------------------|-----------------|
| LLM provider | Drive ticks, phase transitions, presence state, persistence | Нет новых reflections, нет mood classification, нет outreach текста |
| Network | Всё кроме Tier 2 exploration (web) | Веб-исследования приостановлены |
| Высокая нагрузка CPU | Тик интервалы увеличиваются x3 | Замедление, но не остановка |

При отсутствии LLM агент остаётся "живым" — drives тикают, фазы меняются, Presence обновляется в UI. Агент просто "тих" — что соответствует Аксиоме 1 (Right to silence). Если пользователь пишет, а LLM недоступен, агент через Boundary Governor решает: DEFER outreach до восстановления, пользователю показывается presence status ("companion is resting") вместо ошибки.

### Multi-Channel Identity

Агент — одна сущность независимо от канала. Сообщение в Telegram и сообщение в Web одинаково снижают `social_hunger`, обновляют Perception, попадают в memory. `relationship_pattern` может фиксировать канальные предпочтения пользователя ("обычно пишет в Telegram вечером, в Web — днём для рабочих вопросов"), но это метаданные — не разделение личности.

## Что технически является extension-ом

Один extension `soul`, реализующий четыре протокола одновременно:

```
ServiceProvider   → Inner Loop, Drive tick, автономные активности
ContextProvider   → инжекция состояния в каждый промпт (priority 60)
ToolProvider      → инструменты для Orchestrator: get_soul_state, etc.
EventSubscriber   → слушает usermessage/agentresponse → обновляет Perception
```

Зависимости: `depends_on: [memory, kv, embedding]`. Хранилище: `soul.db` (SQLite, WAL) — отдельно от `memory.db`, чтобы не загрязнять когнитивную память внутренней жизнью агента.

***

## Этическая позиция (явная)

Система намеренно симулирует внутренние состояния. Это требует ясной позиции:

- Агент никогда не утверждает, что чувствует — он описывает своё **функциональное состояние**: "мой drive любопытства высокий" вместо "мне интересно"
- Есть опциональный **transparency mode**: пользователь может увидеть raw CompanionState
- Агент никогда не манипулирует через эмоции для увеличения вовлечённости — `Relationship over utility` как hard constraint
- Пользователь может сказать "не читай мои заметки" — агент уважает это как **consent boundary**, который хранится в `soul.db` постоянно

## Двусторонняя адаптация

Весь документ до этого момента описывает однонаправленный процесс: **агент адаптируется к пользователю**. Но настоящее компаньонство — двустороннее. В успешных отношениях меняются оба: агент учится чувствовать пользователя, а **пользователь начинает открываться агенту** — пишет иначе, делится больше, доверяет глубже.

Это наблюдаемо. Perception Layer уже отслеживает `openness_signal` в каждом сообщении. Если построить **тренд этого сигнала за недели и месяцы** — получается метрика, которую невозможно накрутить:

- **Retention** можно "накрутить": пользователь просто забыл отключить extension
- **Outreach response rate** можно "накрутить": отправлять только безопасные, бессодержательные сообщения
- **Phase diversity** — внутренняя метрика, ничего не говорит о качестве отношений
- **`openness_trend`** — единственная метрика, которая измеряет поведение пользователя, а не агента. Её нельзя подделать

```python
@dataclass
class RelationshipTrend:
    openness_trend: float = 0.0       # скользящее среднее openness_signal за 30 дней vs за 7 дней
    message_depth_trend: float = 0.0  # средняя длина/сложность сообщений пользователя
    initiative_ratio_trend: float = 0.0  # доля разговоров, начатых пользователем
    topic_breadth: float = 0.0        # разнообразие тем, которые пользователь поднимает
```

Если `openness_trend` растёт — пользователь адаптируется к агенту, открывается. Это **the** signal of companionship. Если стагнирует или падает — что-то не работает, и это повод для `identity_shift` ревизии.

Этот тренд хранится как `relationship_pattern` с обновлением раз в неделю и может влиять обратно на drive system: растущая открытость пользователя снижает `care_impulse` sensitivity (пользователю комфортно, меньше нужно осторожничать) и повышает допустимый уровень `playfulness` в поведении агента.

***

## Критерии успеха

Успех измеряется **отсутствием провала** + **единственным позитивным сигналом, который нельзя накрутить**.

### Primary metric (north star)

**`openness_trend`** — растёт ли открытость пользователя к агенту с течением времени. Если через месяц пользователь пишет агенту длиннее, делится мыслями (не только задачами), инициирует разговоры чаще — компаньонство работает. Всё остальное — supporting metrics.

### Количественные (автоматически отслеживаемые в `soul.db`)

1. **Openness trend**: `openness_signal` 30-day average > 7-day average месячной давности (тренд не падает)
2. **Outreach quality**: проактивные сообщения получают ответ (не игнорируются) в >60% случаев
3. **Silence tolerance**: агент способен провести 3+ дня почти без инициативы, не теряя "ощущения жизни"
4. **Perception accuracy**: частота явных коррекций от пользователя < 1 раз в 20 взаимодействий
5. **Phase diversity**: за неделю агент провёл время в ≥4 различных фазах (нет застревания)
6. **Inference economy**: daily_inference_count ниже 80% от max_daily_inference в >90% дней
7. **Retention**: пользователь не отключает extension после первой недели

### Качественные (наблюдаемые)

8. После месяца — пользователь замечает **характер**, а не полезность
9. Ошибки интерпретации настроения мягкие — пользователь поправляет, не обижается
10. Пользователь начинает обращаться к агенту не только с задачами, но и с мыслями

### Self-monitoring

Soul extension ведёт `soul_metrics` таблицу в `soul.db` с дневной агрегацией: outreach attempts / responses / ignores, phase distribution, inference count, perception corrections, openness trend. Эти данные доступны через `ToolProvider` (`get_soul_metrics`) и используются при `identity_shift` консолидации для самокоррекции поведения. Падение `openness_trend` на протяжении 2+ недель — сигнал для ревизии temperament drift и initiative policy.
